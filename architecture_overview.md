# 🏛️ AI Job Search Agent Architecture

This document outlines the distributed architecture of the AI Job Search Agent system. The system follows a **Shared-Database Pattern**, where multiple independent services communicate through a common MongoDB instance and Cloudinary storage.

## 🏗️ System Components

### 1. 🚀 API Service (FastAPI)
- **Primary Role:** Handles all user-facing HTTP requests (Chat, Streaming SSE, History, Deletions).
- **Concurrency:** Runs with **N Workers** (e.g., 4) to handle parallel requests.
- **Tools Used:** uvicorn, fastapi.
- **Responsibility:** Triggering real-time agents and serving stored job results from MongoDB.

### 2. 🕒 Scheduler Service (Background Worker)
- **Primary Role:** Manages persistent cron jobs (e.g., "Scrape every 2 hours").
- **Independence:** Runs as a **separate Python process**.
- **Tools Used:** APScheduler with MongoDBJobStore.
- **Responsibility:** Waking up at scheduled times, triggering the agent in the background, and writing results to MongoDB.

### 3. 📦 Shared Persistence Layer
- **MongoDB:** The "Source of Truth". Stores user chat history, conversation summaries, scheduled job metadata, and generated job results.
- **Cloudinary:** Stores physical Excel files generated by both services.

---

## 🛰️ Communication Flow

1. **User Request:** User sends a chat message. The API (FastAPI) handles the message.
2. **Real-time:** The API triggers the Agent immediately for a chat response. 
3. **Persistent Memory:** Both API and Scheduler read/write to the `user_memory` collection in MongoDB.
4. **Scheduled Task:** The Scheduler Service wakes up according to its internal cron (stored in `scheduled_jobs` in MongoDB). 
5. **Background Agent:** The Scheduler triggers the Agent. The Agent uploads an Excel to Cloudinary and logs the result back to MongoDB.
6. **Unified UI:** When the user refreshes their dashboard, the API fetches the latest results from MongoDB (regardless of whether they were from a real-time search or a background schedule).

---

## 🚦 How to Run the System

To handle concurrent requests safely without duplicate cron jobs, you must run **two separate terminal commands**:

### 1. Start the API Server (Multi-Worker)
This runs the web interface.
```bash
# From the project root
uvicorn app:app --host 0.0.0.0 --port 8000 --workers 4
```

### 2. Start the Scheduler Service (Single instance)
This handles the background cron logic.
```bash
# In a second terminal
python scheduler_service.py
```

---

## ✅ Key Implementation Rules
- **No Scheduler in app.py:** The API server should NOT initialize the APScheduler to avoid duplication across workers.
- **State in DB:** Always store agent memory and results in MongoDB so both services can "see" the same user state.
- **Isolation:** If the API server crashes, the Scheduler keeps working. If the Scheduler crashes, users can still chat in real-time.
